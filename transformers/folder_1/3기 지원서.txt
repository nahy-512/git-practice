1. 웹 프로그래밍 팀 프로젝트
 1학년 1학기 때 진행된 웹 프로그래밍 텀프에서 OTT 플랫폼 작품과 설명을 한데 모아볼 수 있는 사이트를 만들었습니다. 저랑 다른 팀원 한 분과 OTT 아이디어가 겹쳐 두 개를 합친 것으로, 넷플릭스, 티빙, 왓챠, 디즈니 플러스 등 OTT 플랫폼은 많은데 정기 구독하지 않으면 해당 플랫폼에 어떤 작품이 있는지 알기 힘들다는 불편함에서 프로젝트를 시작했습니다. 저는 사이트맵 작성, 웹사이트 로고 제작, 넷플릭스 작품 조사와 작업, PPT 작업을 맡았습니다. 아이디어 선정은 각자 문서로 정리한 아이디어 중에서 투표를 진행했는데, 제가 낸 아이디어가 뽑혀 사이트맵을 작성하며 웹사이트 초안을 만들었고, OTT 플랫폼의 로고를 따 저희 사이트만의 로고 작업을 했습니다. 여러 로고 시안을 만들어 팀원들에게 공유하면서 팀원의 의견을 적극적으로 반영했고, 개발 과정에서는 다른 팀원이 막혔던 부분을 넘겨받아 해결하고, 또, Github에 안 올라간다는 팀원의 코드를 넘겨받아 제 것과 합쳐 올렸습니다. PPT는 제가 전반부 작업을 맡았는데, 다른 팀원이 이어서 작업하기 편하도록 템플릿 틀을 먼저 만들어 놓았습니다.

사이트 맵과 로고, PPT 자료 첨부

2. 동아리 활동
 동아리에서 나의 모임 기록 어플 ‘나모’라는 이름으로 프로젝트를 진행했습니다. 프론트와 백엔드가 함께하는 프로젝트는 처음이었는데, 저는 Android 개발자이자 PM으로서 매주 전체 회의와 파트장 회의를 주도했습니다. 서버 회의에도 대부분 제가 참여하긴 했지만, 저도 안드로이드 개발을 진행하면서 프론트 진행 상황은 특히 더 잘 알고 있어서 ‘프론트와 서버의 소통이 잘 안 되는 거 같다.’라는 생각을 항상 저부터가 하게 됐던 것 같습니다. 협업하면서 가장 어려웠던 부분을 꼽자면 단연코 Android와 Spring 간의 소통을 들 수 있습니다. 또한, 이는 제가 가장 신경 썼던 부분이기도 합니다.

 8주간 진행된 프로젝트에서 초반 주차에서는 Android는 UI를 구성하느라, Spring은 테이블을 설계하느라 각자 할 일이 바빠 문제가 없었지만, API 연동 작업이 시작되자 소통 구조에 크게 문제가 있다는 생각이 들었습니다. 때문에 3주차부터는 전체 회의 이후 회고를 도입하면서 개인 작업에 있어서 막히는 부분과 협업에 관한 의견을 들을 수 있었습니다. 4주차 회의 이후엔 서버로부터 Data Grip으로 데이터 확인하는 방법을 배웠고, 노션에 ‘안드-스프링 이슈 소통방’을 만들어 모두가 이슈를 확인하게끔 했습니다.
 저희 팀은 API 명세서를 구글 스프레드시트에서 확인했는데, 작업이 완료됐다고 표시돼 있어 포스트맨으로 확인해보면 오류가 나는 등 서버 쪽에서 작업이 진행 중인지, 완료됐는지를 파악하기 힘들었습니다. 이에 4주차의 회고에서는 제가 낸 ‘작업 완료된 부분이나 오류 있는 부분을 더 명확히 공유할 수단이 필요할 것 같다.’라는 포스트잇이 모든 팀원들의 공감을 얻었고, 좋은 방법이 있을까 얘기하다가 API 명세서를 만든 팀원이 ‘UMC 템플릿을 옮겨올 때, 전체 상황판이 필요할까 싶어서 빼고 만들었는데 그걸 쓰면 될 것 같다.’라고 말해 API 명세서 가장 앞부분에 전체 상황판이 생겨 한눈에 진행 중/이슈/완료 여부를 파악할 수 있게 됐습니다.
